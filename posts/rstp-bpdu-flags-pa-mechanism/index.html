RSTP 知识体系分为四层：端口角色 → 端口状态 → BPDU Flags → P/A 与 TC。角色是"任命"，状态是"现在在干嘛"，Flags 是驱动状态转换的信号。

第一层：端口角色（Port Role）—— 端口在拓扑中的身份#

选举结束后，每个端口被分配一个角色：

角色英文谁拥有朝根方向转发数据
根端口Root Port（RP）每台非根桥有且仅 1 个↑ 朝根✅
指定端口Designated Port（DP）每条网段有且仅 1 个↓ 背根✅
替代端口Alternate Port（AP）下游备用端口↑ 朝根但封住❌
备份端口Backup Port（BP）同一网段备用端口—❌
边缘端口Edge Port（EP）连终端/服务器，管理员配置↓✅

判定逻辑：

  • RP：凡是通往 RB 的通路都是 RP
  • DP：凡是下游有其他交换机的都是 DP
  • AP：凡是通往 RB 但是被阻断的都是 AP
  • BP：凡是交换机自己连自己的都是 BP
  • EP：连接终端且管理员配置的可以成为 EP

AP vs BP 的区分：去掉 RP 和 DP 后，剩下能收 BPDU 的端口——收到别人发出的 BPDU 是 AP，收到自己发出的 BPDU 是 BP。收到自己的报文说明存在自身环路或集线器，前者是错误，后者已被淘汰，因此备份端口实际上很罕见。

线路的封闭是单方面的：一定是上游 DP、下游 AP。每根链路的结构：上游 DP + 下游 RP（主干），或上游 DP + 下游 Alternate（冗余）。

第二层：端口类型（Port Type）—— 端口的链路属性#

类型英文说明
边缘Edge连接终端（主机/服务器），不参与生成树计算
点到点Point-to-Point（P2P）全双工链路，两端各一台交换机
共享Shared半双工/集线器环境，多设备共享同一链路

第三层：端口状态（Port State）—— 端口当前在做什么#

状态英文收 BPDU学 MAC转发数据出现在哪些角色
丢弃Discarding✅❌❌Alternate, Backup, 刚启动的 RP/DP
学习Learning✅✅❌过渡态（P/A 握手和共享链路用）
转发Forwarding✅✅✅RP, DP, Edge

RP 和 DP 不是在 Forwarding 就是在 Discarding（过渡中）。Alternate 永远是 Discarding。

第四层：BPDU Flags 字段 —— 驱动状态转换的信号#

角色和状态是结论，Flags 是推导过程所需的信号。共 8 位：

Bit名称说明
7TCA对 TC 的确认（邻居逐跳可靠握手）
6Agreement下游对 Proposal 的同意
5Forwarding本端口正在转发数据
4Learning本端口正在学习 MAC 地址
3–2Port Role（2 bits）00=Unknown, 01=Alt/Backup, 10=RP, 11=DP
1ProposalDP 请求切换为 Forwarding 状态
0TC（Topology Change）拓扑变更通知

常用 Flags 组合#

场景PropAgreeRoleTCTCA方向
DP 请求握手10DP（11）00DP → RP
RP 同意握手01RP（10）00RP → DP
拒绝（预备端口）00Alt（01）00Alt → DP
拓扑变更通知00任意10全向泛洪
收到 TC 后确认00任意01邻居间

BPDU 发送规则#

所有端口（除 EP）均周期 2s 发送 BPDU。下游靠 BPDU 的 role 信息告知上游自己需要把这条线开启或关闭。

P/A 机制（Proposal / Agreement）—— RSTP 的核心#

RSTP 能做到秒级收敛，关键在于 P/A 机制，全程无计时器参与。P/A 就是让 RP 和 DP 跳过 Listening/Learning 两个过渡态，靠握手直接进 Forwarding。

基本流程#

DP(Discarding) → 发 Proposal(Role=DP) → RP 收到
RP 在新端口上收到 → 同步(其他 DP 切 Discarding) → 回 Agreement(Role=RP)
DP 收到 Agreement → 切 Forwarding
RP 发完 Agreement → 切 Forwarding
  1. 下游交换机 B 的端口角色发生变化（如选举为新 RP），通过 BPDU 的 Port Role 字段告知上游。
  2. 上游交换机 A 检测到变化 → 从该端口下发 Proposal。
  3. B 收到 Proposal，判断该端口是否确认为新 RP。
  4. 若是新 RP，B 立即阻塞连接下游的所有非边缘指定端口，这个过程叫同步。
  5. 同步完成后，B 从新 RP 端口向 A 回发 Agreement。
  6. 双方立即变为 Forwarding 状态。
  7. B 再向下游所有 P2P 端口发送 Proposal，层层递归收敛。

上下游交互规则#

场景行为
上游想与下游建立联系下发 Proposal
下游同意建立联系回发 Agreement
上游切断线路除管理员禁用外不存在此情形；被禁用则沉默等待下游超时
下游切断线路无特殊标记，角色变为 AP 或 DP（后者说明线路颠倒，对端需改角色）

丢包处理#

上游知道下游的 RP 时会一直发送 Proposal，每一个下游 role=RP 都会让上游保持发送 6s。

限制#

共享端口（Shared）不参与 P/A。

拓扑变更机制（TC / TCA）#

TC 是全向泛洪，P/A 是点对点握手，两者解决完全不同的问题。

  1. 发生拓扑变更时，发送 TC。
  2. 发送频率：2s 一次，持续 3s（即发 2 次）。
  3. TC 跟随 hello BPDU 发出，不额外发送 BPDU。
  4. 收到 TCA 后提前结束 TC 发送。
  5. 收到 TC 的邻居回复 TCA，实现逐跳可靠握手。

边缘端口（Edge Port）#

对于连接终端的端口，进行等待确认是多此一举的，因此可以设置为边缘端口。边缘端口检测到接入会直接进入转发模式。

一旦收到 BPDU 报文，端口会瞬间意识到自己其实接在交换系统里，会立刻变成普通接口，要么阻塞，要么进入转发等待期。这种机制比管理员的边缘端口设置命令优先级更高。

计时器#

场景超时时间说明
邻居断联检测6s已连接线路检测对方 hello，6s 没收到认为对方断联
Proposal 超时6s已发送 Proposal，6s 对方没回复，取消此次连接

RSTP 与 STP 混用#

RSTP 能在 RSTP 网络边界端口退回 STP 模式，同时其余部分继续跑 RSTP。
